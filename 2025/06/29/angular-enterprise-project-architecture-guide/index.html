🎯 專案概述

Energy Management System (EMS) Frontend 是一個基於 Angular 19.2.2 的現代化前端應用程式，專為能源管理系統設計。專案採用 Standalone Components 架構、Signals 狀態管理、延遲載入等最新 Angular 技術，並整合 PrimeNG UI 元件庫和 Tailwind CSS 樣式框架。

技術堆疊

  • 框架: Angular 19.2.2 (Standalone Components)
  • 狀態管理: Angular Signals
  • 路由: Angular Router (延遲載入)
  • UI 元件庫: PrimeNG 19.0.10
  • 樣式框架: Tailwind CSS + PrimeFlex
  • 構建工具: Angular CLI
  • 程式碼品質: ESLint + Prettier
  • 伺服器端渲染: Angular SSR

專案特色

  • ✅ 現代化技術棧: 採用 Angular 19 最新特性
  • ✅ 模組化架構: 清晰的功能模組劃分
  • ✅ 效能優化: 延遲載入與 OnPush 策略
  • ✅ 開發體驗: 完整的工具鏈與程式碼規範
  • ✅ 可擴展性: 支援快速功能模組擴展

🏗️ 整體架構設計

架構模式

本專案採用 功能模組化架構 (Feature-Module Architecture)，結合 清潔架構 (Clean Architecture) 原則：

分層責任與設計原則

層級 責任範圍 設計原則
Core Layer 核心業務邏輯、服務、攔截器 提供基礎設施，保持單例模式
Shared Layer 可重用元件、管道、指令 跨模組共享，無業務邏輯依賴
Feature Layer 功能模組、業務邏輯 自包含，延遲載入，依賴隔離
Page Layer 頁面級元件 整合功能模組，處理全域邏輯

架構優勢

  1. 清晰的職責分離: 每層都有明確的責任範圍
  2. 良好的可維護性: 模組化設計便於維護和擴展
  3. 優秀的效能表現: 延遲載入與變更檢測優化
  4. 團隊協作友好: 不同功能可並行開發

🏗️ 應用程式核心結構

層級 資料夾 目的 責任範圍
核心層 core/ 🔧 基礎設施提供 核心服務、資料模型、攔截器、路由守衛等應用程式基礎功能
共享層 shared/ 🤝 可重用資源 跨模組共享的元件、指令、管道、工具函式與版面配置
功能層 features/ 🎯 業務功能實現 獨立的業務功能模組，包含完整的元件、頁面、服務與路由
頁面層 pages/ 📄 頂層頁面整合 應用程式層級的頁面元件，整合多個功能模組

🧱 核心架構層級詳解

🔧 Core 層 (src/app/core/)

責任範圍:

  • enums/: 定義應用程式中使用的列舉類型
  • guards/: 路由層級的權限控制與導航守衛
  • interceptors/: HTTP 請求/回應的攔截處理
  • models/: TypeScript 介面與資料模型定義
  • services/: 單例服務，處理核心業務邏輯

設計原則:

  • Core 層應為應用程式提供基礎設施
  • 服務應使用依賴注入並保持單例模式
  • 遵循單一責任原則

🤝 Shared 層 (src/app/shared/)

責任範圍:

  • components/: 跨功能模組重用的 UI 元件
  • directives/: 自訂行為指令
  • layouts/: 頁面版面配置元件
  • pipes/: 資料轉換管道
  • utils/: 純函式工具庫

🎪 功能模組內部結構

每個功能模組 (features/*/) 都遵循統一的內部結構：

子資料夾 目的 說明
components/ 🧩 模組專用元件 僅在該功能模組內使用的 UI 元件
pages/ 📄 模組頁面元件 該功能模組的各個頁面元件
services/ ⚡ 模組業務邏輯 處理該功能模組的資料存取與業務邏輯
routes.ts 🛣️ 模組路由配置 定義該功能模組的路由結構與導航

模組設計原則:

  1. 自包含性: 每個功能模組包含完整的業務邏輯
  2. 延遲載入: 使用 loadComponent 實現按需載入
  3. 路由分離: 每個模組維護獨立的路由配置
  4. 依賴隔離: 避免功能模組間的直接依賴

📄 Pages 層詳解 (src/app/pages/)

責任範圍:

  • 提供頂層頁面元件
  • 整合多個功能模組
  • 處理全域頁面邏輯

🎨 版面配置系統

版面配置 檔案位置 適用場景 特色功能
空白版面 shared/layouts/empty-layout/ 登入頁面、錯誤頁面 極簡設計、全螢幕佈局、內容投影
側邊欄版面 shared/layouts/sidebar-layout/ 主要應用頁面 響應式導航、暗色模式、收縮側邊欄

🔄 開發工作流程目錄使用指南

  1. 新功能開發: 在 features/ 下建立新的功能模組
  2. 共用元件: 開發可重用元件放置於 shared/components/
  3. 核心服務: 應用程式級服務放置於 core/services/
  4. 資料模型: TypeScript 介面定義於 core/models/
  5. 工具函式: 純函式工具放置於 shared/utils/

🏗️ 架構優勢與設計決策

技術優勢

  1. 現代化 Angular:

    • Standalone Components 減少 NgModule 複雜性
    • Signals 提供更高效的響應式狀態管理
    • 函式式路由守衛提高可測試性

  2. 效能優化:

    • 延遲載入減少初始包大小
    • OnPush 變更檢測策略
    • Deferrable Views 改善載入狀態

  3. 開發體驗:

    • TypeScript 嚴格模式確保類型安全
    • ESLint + Prettier 自動化程式碼品質
    • 清晰的資料夾結構便於導航

  4. 可維護性:

    • 功能模組化促進程式碼重用
    • 清潔架構分離關注點
    • 完整的文件與註解

設計決策理由

  1. 為何選擇 Standalone Components?

    • 減少 NgModule 樣板程式碼
    • 更好的 Tree-shaking 支援
    • 簡化測試設定

  2. 為何採用功能模組架構?

    • 促進團隊協作，不同功能可並行開發
    • 延遲載入優化效能
    • 便於功能的增減與維護

  3. 為何整合 PrimeNG?

    • 豐富的 UI 元件庫
    • Angular 原生支援
    • 一致的設計語言

🚀 最佳實踐建議

開發規範

  1. 元件設計:

    • 保持元件單一責任
    • 使用 OnPush 變更檢測
    • 實施適當的生命週期管理

  2. 狀態管理:

    • 優先使用 Signals
    • 複雜狀態考慮 NgRx
    • 避免過度的狀態提升

  3. 效能優化:

    • 使用 TrackBy 函式優化列表渲染
    • 實施虛擬滾動處理大量資料
    • 適當使用快取策略

擴展指導

  1. 新增功能模組:

  2. 共享元件開發:

    • 放置於 shared/components/
    • 提供清晰的 API 介面
    • 撰寫使用文件

  3. 服務層擴展:

    • 核心服務放置於 core/services/
    • 功能專用服務放置於各自模組
    • 遵循依賴注入最佳實踐
